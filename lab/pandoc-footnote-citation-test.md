# Test brief: Pandoc citeproc footnotes vs. hand-written GFM footnotes

## 1. Context

BlobTxt relies on GFM-style footnotes:

```markdown
This is some body text[^2]

[^2]: And this is a reference definition.
```

Another thing planned is citing sources using Pandoc's citation syntax: (`[@key]`), with bibliographic data coming from a CSL-JSON or BibTeX file (generated by Zotero's Better BibTeX plugin, though that detail doesn't matter for this test).

The question: when using a *notes-based* CSL style (e.g. Chicago Notes-Bibliography, as opposed to Chicago Author-Date), Pandoc's citeproc converts inline `[@key]` citations into generated footnotes. I need to know exactly how those generated footnotes interact with my own hand-written `[^N]` footnotes in the same document. Specifically:

1. Do the two footnote "streams" (mine vs. citeproc-generated) get merged into a single, correctly renumbered sequence in the output?
2. Or do they end up as separate, non-interleaved numbering sequences?
3. Does ordering in the source document determine final numbering, or does citeproc do something order-independent?
4. Does anything break, duplicate, or silently drop if a hand-written footnote and a citation-generated footnote land on the same line/sentence?

I could not find official Pandoc documentation that addresses the *merge-with-pre-existing-footnotes* case directly, only that citeproc converts citations into footnotes under notes-styles. This test is meant to close that gap empirically.

## 2. Plans
### 2.1. Prepwork

Record the version (`pandoc --version`) used for testing. Citation/citeproc behavior has changed across Pandoc versions (the old standalone `pandoc-citeproc` filter is deprecated in favor of the built-in `--citeproc` flag in modern Pandoc).

Create a minimal CSL-JSON bibliography file (`refs.json`) with **two** fake but valid entries, so multiple citations can be tested. Made-up entries will suffice.

Create a test Markdown file (`test.md`) that includes:

1. At least 3 hand-written `[^N]`-style footnotes, placed at different points in the document (beginning, middle, end).
2. At least 2 Pandoc citations (`[@key1]`, `[@key2]`) interspersed among the hand-written footnotes (not all bunched together).
3. One sentence that has **both** a hand-written footnote marker and a citation immediately adjacent (to test the "same sentence" edge case), e.g.: 

```markdown
This is a claim with my own aside[^1] and also a citation [@key1].
```

4. YAML frontmatter with `bibliography: refs.json` and a notes-based CSL style. A notes-based CSL file will be needed. Chicago Notes-Bibliography (`chicago-fullnote-bibliography.csl` or similar) is the standard test case. It's available from the CSL style repository (github.com/citation-style-language/styles).

### 2.2. Test

Run:
```bash
pandoc test.md --citeproc -o test.html
```

Use `.html` output first since it's easiest to inspect. Inspect the output and report:

1. The final footnote numbering order (does it match source order, or is it reordered?).
2. Whether hand-written and citeproc-generated footnotes are indistinguishable in the output (same numbering sequence, same visual style), or visibly separate.
3. What happened in the "same sentence" edge case specifically; did both notes render correctly, in what order, under what numbers?
4. Any errors, warnings, or unexpected behavior from Pandoc during the run.

### 2.3. Control

As a control, repeat the same `test.md` but swap the CSL style to an **author-date** style (e.g. `chicago-author-date.csl`) and confirm that in this case citations stay inline (no footnotes generated for them) and the hand-written `[^N]` footnotes are completely unaffected. This confirms the author-date path is "safe" as expected, as a baseline comparison against the notes-style behavior above.
